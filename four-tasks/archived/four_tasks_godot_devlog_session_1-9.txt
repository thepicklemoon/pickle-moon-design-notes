THE APPTRIOC FORK ARCHITECTURE (locked session 5):
  APPtrioc is no longer a marketing fork of the prototype. It is a
  Godot HTML5 export from a snapshot of Four Tasks at the moment
  tile 4.14a is closed (basic picker, pre-context-menu). The fork
  point sits between tile 4.14a (pre-fork, APPtrioc inherits) and
  tile 4.14b (post-fork, Four Tasks exclusive — theme depth
  features). Launch is SEQUENCED, not simultaneous: Four Tasks ships
  to stores first (Phase 5), APPtrioc is built+deployed afterward
  (Phase 5b), the atrioc video drops once both are live.
  See top of `four_tasks_godot_todo.txt` for the full fork
  architecture and launch sequence. Both backends (Four Tasks +
  APPtrioc) run on Cloudflare in separate Workers + D1 instances
  in the same Cloudflare account — Firebase considered and rejected
  for APPtrioc; separate-Workers-in-one-account gives isolation
  without the two-vendor tax.

Top-level scene tree:
  Main (Node)
  ├── Backend (Autoload — singleton)        — server I/O, signals
  ├── State (Autoload — singleton)          — in-memory mirror of pair data
  ├── ThemeManager (Autoload — singleton)   — palette swap, theme_changed signal
  ├── Audio (Autoload — singleton, v1.x)    — slot ticks, coin sounds, stamp thump
  ├── Loader (Scene)                        — identity check, pair-key resolve
  ├── Onboarding (Scene)                    — name, tasks, slot intro, emoji pick
  ├── App (Scene)                           — main two-panel UI
  └── DevKit (Scene, debug-only)            — bug inbox, seed tools

Source of truth: server. State is a mirror.
Write flow: UI → State.set_*() → Backend.save() → server confirms
            → State emits signal → UI updates from signal.
Optimistic local updates with rollback on server reject.

Server architecture (locked session 2):
  - Cloudflare Worker (TypeScript) at four-tasks-api.thepicklemoon.workers.dev
  - D1 SQL database (Oceania region) bound as env.DB
  - Four tables: pairs, users, days, bug_reports
  - Pair-key migrations are re-entrant + transaction-wrapped

DEV ENVIRONMENT LESSONS:
  - Windows PowerShell `curl` is an alias for Invoke-WebRequest, NOT
    real curl. Different semantics, security prompt UX, different
    output format. Adopted convention: use `curl.exe` explicitly on
    this project, always. Same flags as everywhere else, same output
    as everywhere else.
  - PowerShell mangles JSON bodies on the command line — backslash-
    escaped quotes don't survive interpolation. Stored JSON in a file
    and used `--data-binary "@file.json"` to sidestep quoting. Adopted
    as the convention for any curl-with-body command going forward.
  - `npx wrangler dev` drifted to remote mode mid-startup twice during
    this session (no clear root cause — possibly hotkey, possibly
    config). Workaround: `npx wrangler dev --local` with the explicit
    flag prevents the drift. Adopted as the convention.
  - Local D1 is a separate SQLite file from remote D1. Tile 1.1 only
    applied schema to remote. First time `--local` ran, `pairs` table
    didn't exist locally and the endpoint 500'd. Fix: apply schema.sql
    with `--local`. Migrations 001/002 are baked into schema.sql so
    re-applying them to a fresh local DB fails with "duplicate column"
    — harmless, skip them for local seed. Will document the fresh-
    clone setup sequence in the /server/README.md when that lands in
    tile 1.3.
  - SQL seed via shell: PowerShell escaped-quote interpolation
    corrupted a JSON string column on first insert attempt (stored
    `\workout\` instead of `"workout"`). Caught immediately because
    the endpoint 500'd on JSON.parse — the try/catch did exactly what
    it was designed for. Fix: put SQL in a .sql file, use `--file=`,
    no shell escaping in play. Convention adopted: anytime SQL needs
    quoted strings, write the SQL to a file.

  Pattern across these: PowerShell's argument handling is a
  consistent source of friction. The fix is the same every time —
  use an explicit executable (curl.exe), use file references over
  inline strings (--data-binary @file, --file=seed.sql). Doing this
  by default removes a class of mystery bugs.

  MIGRATION LOGIC — the dense part of PUT user:
  - When username or emoji changes, the pair-key changes.
    Migration is the re-entrant, transaction-wrapped flow already
    sketched in v2 design notes. This session locked the response
    shape, collision handling, and client-side defensive write
    requirements.
  - Atomic transaction inside a D1 batch(): re-derive new key,
    INSERT new pair row, INSERT both users under new key, COPY
    all days rows under new key, DELETE old days/users/pair rows.
    Insert-new-then-delete-old rather than UPDATE pair_key in
    place — D1 (SQLite) doesn't handle primary-key column updates
    cleanly; copy-then-delete is the natural fit.
  - Response shape locked: always includes `migrated` boolean,
    always returns full pair payload. When migrated: true, also
    includes new_pair_key. Same shape regardless of path so the
    client always branches on data.migrated, no special case for
    "did the response include a new key?"
  - Collision (1-in-5-billion hash collision): server lets the
    INSERT into pairs fail naturally with primary-key violation,
    transaction rolls back, returns 409 pair_key_collision.
    Client surfaces as "that username is taken, sorry try
    another" — same UX as onboarding collision. No pre-emptive
    check. Discussed the information-leak angle: pre-checking
    would leak "this exact six-tuple combination is taken," but
    the leak is near-zero since the attacker would need the
    partner's full six-tuple to construct a collision, at which
    point they already have recovery credentials. Decision was
    "let it fail" for simplicity, not security.
  - PARTNER EXPERIENCE during migration is silent and automatic.
    Their next poll on the old key returns 404, client recovery
    path re-derives the new key from the six values they still
    hold (five visible on screen + own name), client updates
    stored identity and re-polls. No popup, no error, just a
    re-themed partner panel. The 404 recovery is the same code
    path that handles any stale-key situation, not migration-
    specific.
  - CLIENT-SIDE DEFENSIVE WRITES required to avoid lockout from
    in-flight failure. The conversation surfaced four
    requirements: persist pending change to identity.cfg before
    sending, use pending values during 404 recovery, clear
    pending slot on confirmed success, retry against new key
    on concurrent-migration race. These are picked up by tile
    1.7 (Godot identity layer) and tiles 4.14 / 4.15 (emoji and
    username edit flows). Captured in design doc as a section.
  - NICE-TO-HAVE captured for tile 4.15: small dismissible
    popup on username change ("changes may take a minute to
    take effect", "don't show again"). UX layer over the
    defensive writes, not a correctness layer.

  THE PALETTE-AGNOSTIC ART RULE:
    All theme art (including the sticker itself) painted in a
    canonical reference palette. Each sticker ships palette.tres
    mapping reference colours to that sticker's natural-state RGB.
    At render time, the engine substitutes the reference palette
    for the active palette source's tagged colours.
    Consequences: one source of truth per sticker (no "sticker
    frog" + "theme frog" duplication); cross-theme clash solves
    itself (lily pads retint when vampire becomes palette source,
    no bright-green-on-black-and-red clash).
    Exception: the picker grid renders each sticker in ITS OWN
    palette, not the active palette source's, otherwise every
    sticker looks identical in the picker. Picker preserves
    visual identity for choosing.

  FILE ARCHITECTURE — FOLDER PER STICKER:

    sticker capability by which optional files exist. No separate
    manifest.

EIGHT DESIGN QUESTIONS LOCKED:

  Q1 — THE MOTD STRUCTURE.
    The MOTD is FOUR spinners (adverb + verb + noun + emoji),
    not three. The emoji slot reads from the SAME pool the user
    has curated as Lever 1 pool membership in the sticker
    picker. Single source of truth — change your pool, your MOTD
    emoji vocabulary changes with it.
    PROTOTYPE BUG: spinners alternate visibly rather than
    randomising truly. Fix at tile 4.4 implementation (true
    independent randomisation per slot).
    SUPERSEDES: implicit assumption in theme doc that the pool
    was only for day-cell stickers. Theme doc now updated to
    note the dual purpose.

  Q2 — CRASH RESISTANCE.
    The morning sequence is PURELY COSMETIC. Server-side, coins
    are updated as a number by the nightly cron and the claim
    endpoint, not derived from animation events. Crash mid-
    sequence has three cases:
      (a) Before claim fires: replay on next open. No loss.
      (b) After claim, before animation completes: skip on next
          open (server flag cleared). User missed the ceremony
          but data correct.
      (c) Network/server failure on claim: retry on next open
          via still-set flag. Robust.
    The "infinite coins" failure mode is structurally impossible
    because the claim endpoint is atomic and idempotent.

  Q3 — BASE STATE IS ALWAYS CALENDAR-FLAT.
    When the user opens the app, the first frame is the calendar
    in its normal state. No trays open, no animations in progress.
    Then the morning sequence fires (if a payout is pending) as
    an OVERLAY-ON-TOP, not as a boot mode.
    Implications: Loader scene (tile 2.1) doesn't check for
    pending payout — that's App scene's job after it boots. Same
    code path always; sequence is an interaction layered on the
    calendar.

  Q4 — PARTNER PANEL VISIBILITY MODEL.
    Your morning sequence is PRIVATE. Partner doesn't see it.
    More importantly: the partner panel reflects YOUR yesterday
    as UNSTAMPED until you've claimed. Your claim is the moment
    state becomes visible to your partner.
    The "dead until they've claimed" property is intentional
    design, not a limitation. Makes the connection feel alive —
    state moves when partner moves it, not when the server
    seals it. The textural moment of "I see their stamp appear
    live on my partner panel because they just opened their app"
    is the connection mechanic. iMessage-like quality.
    Polling (tile 2.8, 30s cadence) catches the change. Tile
    2.8 must refresh day-cell state, not just metadata.
    CRITICAL BUG TO AVOID: the prototype had a "prestamped"
    bug where days appeared stamped before claim. Morgan
    identified and fixed. The Godot port enforces this through
    the claim endpoint being the moment of state-becomes-visible.

  Q5 — NO MULTI-DAY BACKLOG.
    "What if user is away a week?" → at most ONE pending payout.
    Data model enforces it for free: payout requires tasks_done
    > 0, which requires the app open that day, which requires
    the user's engagement. Days between the last-played day and
    today are EMPTY days — no payout earned, no ceremony due.
    The walkback algorithm finds the most recent unclaimed-but-
    completed day and plays sequence for that. Streak breaks
    naturally during absence (no four-of-four days), resets
    silently — no special "you broke your streak" announcement.

  Q6 — REST DAY VARIANT.
    Rest day plays the SAME morning sequence. NOT a separate
    sequence. Same beat structure, same timing, same lockout —
    just different content slots:
      - Tasks typewriter-glitch into rest-vocabulary phrases
        ("relaxed", "had a cheat meal", "unworried", etc.) over
        the user's original labels.
      - Checkboxes get purple ticks/hearts instead of green/red.
      - No coins fly out (rest cost was paid in advance at
        rest-day designation, tile 4.8).
      - Stamp is purple with a rest-pool message ("good on you
        for the rest", "earned", "well-deserved", "soft day,
        well chosen").
    Single parameterised animation — cleaner than two parallel
    sequences. New content pools needed: rest-day task labels
    (~15-30 phrases) and rest-day stamp messages (small pool
    with affirmation tone).

  Q7 — PARTNER REACTION SURFACING.
    Reactions are TIME-AGNOSTIC, not event-driven. Surface via
    benign visual indicators on past-day cells in BOTH
    directions. No push notifications, no animation, no urgent
    badges.
      Direction 1 (your view of partner's panel): partner's
        sealed days you haven't reacted to yet get an ambient
        marker. Clears on react-or-skip.
      Direction 2 (your view of your own panel): your past-day
        cells that received a reaction get a marker drawing
        eye to them. Clears on view.
    Same indicator pattern in both directions. Symmetric
    vocabulary.
    Tutorial reveal at day 4 (preferred) or day 6 (backup) per
    staggered disclosure schedule. Discovery is taught BEFORE
    a reaction necessarily arrives, so users have the vocabulary
    to notice the indicator when one does.
    SUPERSEDES: four_tasks_partner_reactions_design_notes.md
    "Surfacing reactions to the receiver" section which had
    described push-based surfacing. Push is no longer the
    model.
    SUPERSEDES: staggered disclosure framing of partner
    reactions as purely trigger-gated. Scheduled day-4 reveal
    is now the primary teaching moment; trigger-gated reaffirm
    can fire if a reaction lands before that.

  Q8 — MOTD REROLL COST.
    Substantive change from prototype.
    PROTOTYPE: day-1 free, day-2+ DOUBLING cost from 1 coin.
    PROBLEM: doubling uncapped means user not paying attention
    could blitz through tens of thousands of saved coins in
    ~12 presses (cost goes 1, 2, 4, 8, 16... reaches 4096 at
    press 13). A casual aesthetic preference becomes a major
    economic event. Wrong mechanic for casual reroll.
    LOCKED: flat random range 90-110 coins per press, no
    escalation. Approximately 1/10 of an unmultiplied daily
    task payout. The 90-110 range (vs flat 100) creates a
    small textural moment of uncertainty per press.
    Architectural principle that fell out: doubling-cost
    belongs on features where the user output should ESCALATE
    in significance (coin name reroll = persistent identity,
    keep doubling). Flat-cost belongs on features that should
    stay CASUAL (MOTD reroll = transient aesthetic, drop
    doubling).
    MOTD reroll is a coin economy VENT, not a sink. Real sinks
    are emoji packs (planned 1-3x rest day cost) and rest days
    themselves.
    SUPERSEDES: tile 4.5's prior description (doubling cost)
    and the prototype's mechanic.

KEY ARCHITECTURAL CALLS THIS EXTENDED SESSION:

  1. The morning sequence is an overlay-on-top of calendar
     base state, not a boot mode. Same code path always.

  2. Server is the source of truth for coin balance, stamp,
     streak; animation is the user's EXPERIENCE of these, not
     the data itself. Decouples robustness from animation
     correctness.

  3. Claim endpoint atomically updates server state at the
     START of the sequence. Server response drives the rest
     of the animation. Animation never plays content the
     server might disagree with.

  4. Partner panel state lags behind the user's own panel by
     "until the user claims." This is connection-making, not
     latency.

  5. Rest day = same sequence, parameterised content. One
     animation maintained, not two. Clarity over cleverness.

  6. Time-agnostic features (partner reactions) get ambient
     visual indicators, not push or animation. Surface
     vocabulary follows feature rhythm.

  7. Doubling-cost vs flat-cost reroll mechanic is determined
     by whether the feature should feel ESCALATING vs CASUAL.
     This is now a general principle in the project.

  8. Migration time budget is the morning sequence duration
     (~15-30s), not "indefinite." Real upper bound. Operations
     that exceed it need to split pre/post-claim or accept
     follow-up sync.

  LOCK 1 — TIMEZONE LOCALISATION + LAZY SEAL ON OPEN
  (four_tasks_timezone_and_sealing_design_notes.md, new):

    Reframes tile 1.4 entirely. The prior "Worker scheduled cron —
    nightly UTC sweep" model doesn't survive localisation —
    sealing yesterday at UTC midnight fires at the wrong wall-
    clock time for any user outside UTC±a-few-hours. Globally
    available app needs per-user day boundaries.

    Decision: each user has stored IANA timezone (e.g.
    "Australia/Perth"). Mutable. Not part of pair-key hash.
    Migration_004 adds users.timezone TEXT DEFAULT 'UTC'.
    Onboarding captures device tz silently. Boot-time tz check
    fires PUT user if device tz differs from stored.

    Sealing becomes LAZY. No cron job. Seal logic lives inside
    the claim endpoint transaction (tile 1.3 territory). When a
    user opens the app and the claim endpoint fires, any
    unsealed days <= local_date - 1 get sealed in the same
    transaction. User offline for three days → server walks the
    gap on next open (bounded work — even a year offline is 365
    row updates).

    Client sends local_date on every relevant write. Server
    validates plausibility against stored tz (±4hr window).
    Single source of truth: the user's lived reality, told by
    the client, sanity-checked by the server.

    Cross-user interaction stays minimal. Each user's seal is
    their own concern only. Partner panel renders truthfully
    from whatever's in the DB — if their day rolled and they
    haven't opened the app, panel shows their incomplete day
    frozen. The 1.5x multiplier check is a read, no side effect.
    No bridging code beyond what already exists.

    Non-goal: anti-date-scrubbing. The morning_payout_due_for
    column already makes the payout a single-use voucher.
    Rolling clock forward skips days, doesn't farm them. No
    leaderboard planned, removing the primary motivation. The
    cost of pretending to defend against clock manipulation is
    UX friction (false-positive rejections for legitimate
    timezone changes); the cost of ignoring it is zero in the
    product as designed.

    DST handled by IANA names via a date library (Luxon
    recommended — runs in Cloudflare Workers' modern V8, handles
    DST transitions correctly). Don't roll your own.

  LOCK 2 — ONBOARDING WITH SOLO MODE + INVITE LINK
  (four_tasks_onboarding_design_notes.md, new):

  LOCK 3 — STICKER CANVAS SIZE
  (four_tasks_theme_design_notes.md update):

    Morgan asked about getting Aseprite work started at FIFO
    shifts. Needed a locked canvas size before painting. First
    pass recommended 256×256 source for downscale-and-paint
    workflow — Morgan correctly flagged this as "disgustingly
    huge" given the pixel-art aesthetic the rest of the project
    has been pointing at.

    Re-derived from the right side of the question. Pixel-art
    workflow paints AT logical pixel size, lets Godot's Nearest
    filter scale up. Not high-fi paint-and-downscale.

    Math (re-done correctly):
      - 1080×2400 portrait viewport.
      - 7-column calendar, ~12px padding, 6×6px gaps → cell
        ~134px wide at native resolution.
      - Sticker renders at ~75% of cell width (prototype ratio)
        → target on-screen size ~100px.
      - At 32×32 source → ~3x integer scale on cell, ~6x scale
        in picker grid. Clean Nearest, no fractional artefacts.

    Locked: 32×32 RGBA, transparent background. Effective art
    area ~26×26 inside with margin for rotation wobble (the
    prototype's stickerOffset rotate() applies on day cells).

    Reasoning for the upper end of pixel-art viability (32 vs
    16/24):
      - Detail headroom for a novice pixel artist. Easier to
        simplify a 32 down to 24 later than add detail to a 16.
      - Future "chunkier, more stylised" releases can re-export
        at 16 or 24 from 32 sources; forward-compatible.
      - Picker grid render needs the sticker to SELL its theme;
        32×32 at 6x upscale fills the picker confidently.

    Practical Aseprite setup captured in part 2 conversation:
    Indexed mode recommended (option C for retint mechanism
    leaves cleanest), three palette slots reserved for themed
    roles (primary/secondary/tertiary) painted in reference
    values at export.

        background, cell variants, dead-cell, grid_tile,

        negotiate.
      - The user's library is a menu of options per slot.
      - Instant feedback on slot change — no commit-on-close
        pattern (that exists for identity, doesn't apply here).
      - Design sparsity is its own axis. A minimal sticker
        with one exquisite element can be more compelling than
        a maximal one with all of them.

        what makes sense for it, no "TIER 1 bar" forcing
        completeness per pack.

  LOCK 1 — PIXEL-FREQUENCY PALETTE DERIVATION
  (four_tasks_theme_design_notes.md — major rewrite):

    The architectural shadow of the day. Started as a workflow
    question ("how do I include palette data alongside the
    sticker"), ended as a simplification of the previous
    session 7 model.

    Previous session 7 model:
      Each sticker ships sticker.png + palette.tres.
      Art painted in canonical reference colours
      (#FF0000/#00FF00/#0000FF as primary/secondary/tertiary
      role-values). palette.tres declares real colours for
      each role. Renderer substitutes reference → real at draw
      time for every themed surface (including the sticker
      itself).

    Morgan's proposal:
      Drop palette.tres entirely. Paint stickers in their REAL
      colours (frog as actual mossy green, etc.). At runtime,
      the engine reads sticker.png pixels, counts colour
      frequencies, derives primary/secondary/tertiary from the
      pixel counts (most-painted = primary, etc.). Black, white,
      and greys excluded from the count entirely (they pass
      through unchanged as structural — outlines, shadows,
      highlights).

      Constraint: each sticker uses no more than three non-grey
      colours, with clear proportional difference (no near
      50/50 splits).

    Claude initially pushed back on multiple grounds:
      - Conflates "most painted" with "most important."
      - Removes authorial control over which colour becomes
        which role.
      - Edits to art silently flip ordering.
      - Can't express "this colour is purely structural."
      - Debuggability suffers (algorithm vs file).

    Morgan rebutted by clarifying the proportional-difference
    constraint addresses the ambiguity concern; the B/W/grey
    exclusion handles the structural-colour concern; the
    authorial control is preserved by painting (the painter
    expresses palette intent by how much area each colour gets).
    Asked Claude to reconsider rather than concede.

    On reconsideration: Morgan's right. Under the painter's
    constraint, the painted PNG already unambiguously expresses
    primary/secondary/tertiary. A sidecar palette.tres becomes a
    redundant second source of truth that can drift from the art
    when edits happen. Removing it eliminates a class of bugs
    (art/metadata sync) and one piece of per-sticker authoring
    overhead.

    The wizard counterexample Claude had raised doesn't survive
    the painter's constraint: if you want purple as primary you
    paint purple as the dominant area. The choice of WHAT AREA
    each colour occupies IS the authorial control. You're not
    losing control — you're encoding it in the painting itself
    rather than in a sidecar.

    LOCKED SESSION 8: palette derivation by pixel-frequency
    counting from sticker.png. No palette.tres file. Sticker.png
    is single source of truth for both leader visual AND
    palette declaration.

    Sticker.png is the explicit exception to the
    palette-agnostic art rule — it's painted in REAL colours
    (because it IS the palette declaration), not in reference
    role-values. All OTHER themed surfaces (cell overlays,
    backgrounds, grid tiles, flourishes, label backgrounds,
    effect sprites) remain painted in reference role-values and
    retinted at runtime, exactly as the previous model
    specified.

    Graceful failure: if a sticker uses more than three non-grey
    colours, the first three by pixel count become the roles;
    the rest still render as painted in the sticker but don't
    propagate to themed art. No load-time validation. The
    constraint is a discipline; the devtool (below) catches
    mistakes.

  NEW SECTION — PALETTE DERIVATION DEVTOOL:

    Surfaced during the palette-system conversation: under the
    new rule, the painter needs to know which colour will become
    which role BEFORE shipping. Without a visualiser, the
    options are manual pixel-counting in Aseprite or
    ship-and-test in the live app — both clunky.

    Spec captured: dev-only utility that loads a sticker.png,
    displays the derived palette swatches (primary/secondary/
    tertiary) with RGB values + pixel counts, shows all
    non-grey colours sorted by frequency, badges the result
    when adjacent ranks are within tolerance (starting
    threshold 10% pixel-count difference).

    READ-ONLY by design. The devtool cannot write a sidecar,
    cannot override the derived ordering, cannot produce any
    artefact other than displayed information. If the
    derivation produces the wrong ordering, the fix is to
    re-paint — adjust pixels in Aseprite until the desired
    ordering emerges. There is no "the algorithm says X but I
    want Y" escape hatch, because that hatch would reintroduce
    the drift problem the no-sidecar rule was designed to
    prevent.

    Lives in scenes/devkit.tscn (tile 0.4 currently a sanity-
    check skeleton, fleshed out by tile 5.2). Pre-tile-5.2 it
    can run standalone via F6.

    Implementation note captured: the grey-exclusion rule used
    in the devtool MUST match the rule used at runtime in the
    actual renderer. Shared derivation function. Implementation
    order: write the function, use it in devtool, validate by
    eye, then wire into runtime renderer.

  LOCK 1 — SOLO MODE IS THE DEFAULT OUTCOME OF ONBOARDING
  (four_tasks_onboarding_design_notes.md, rewrite):

    The session 6 onboarding doc treated solo and paired as
    two parallel flows with a fork screen. Roleplay surfaced
    that the fork is doing real damage — it forces a
    relational question at the front door of the app.

    Reframe: solo IS the default. There is no fork. Every
    user finishes onboarding in solo mode. Pairing is a
    POST-ONBOARDING action via a persistent "Add my partner"
    button on the always-empty partner calendar.

    Architectural consequences:
      - The invite link flow doesn't disappear — it's now a
        deep-link target that opens the existing solo user's
        app at the "Add my partner" interaction, prefilling
        the inviter's values.
      - Manual fallback is the post-onboarding "Add my
        partner" button — no longer two flow paths through
        onboarding, just one flow into solo + one button
        afterward.
      - "Re-derive solo mode from the user-needs angle"
        observation from session 6 is now LOAD-BEARING, not
        a graceful concession.

    Why this is better than session 6's fork model:
      1. Removes the "are you alone? answer publicly" moment
         at install time. Friction-cliff for any user who
         doesn't know yet what their partner will say to the
         install pitch.
      2. Single flow simplifies the entire tile structure of
         Phase 3 — no branch tiles, no fork-screen tile, no
         invite-link-detection tile inside onboarding.
      3. Empty partner calendar IS the recruitment surface
         (already established in session 6). The persistent
         "Add my partner" button is the natural surfacing of
         that surface.

  LOCK 2 — PAIRING BY TYPED THREE VALUES
  (four_tasks_onboarding_design_notes.md, rewrite):

    Considered two alternatives to "type the three values"
    pairing:
      (a) Server-issued opaque codes. Joiner types a 6-char
          code, server resolves to inviter.
      (b) Pre-computed hash blobs. Joiner types a long hash
          string carrying full inviter identity.

    Both rejected:
      - (a) routes the relationship through a system
        identifier (the code). Feels like a customer-service
        ticket, not a partnership. Wrong tone for the buddy-
        ware positioning.
      - (b) creates an oracle concern (any hash-blob lookup
        endpoint is a stand-alone identity verifier) and
        forces UX around copy-paste of an ugly string.

    Locked: typed three values (name + username + icon), same
    as the inviter's. The values are already on the inviter's
    screen by design — pair-key v2's recovery story works
    because of this. The joiner types what they see.

    Ambiguous-match resolution:
      Two unrelated users with identical (name, username, icon)
      tuples is plausible at scale. Server returns 409 on
      ambiguous match. Joiner-side copy asks the INVITER to
      vary their username temporarily — joiner connects, then
      inviter reverts. Full pair-key hash includes both halves,
      so once paired the username can revert without identity
      disturbance.

    This required pair-key migration logic to support
    post-pair username revert as a supported, non-extraordinary
    flow. See LOCK 4 below.

    SECTION 4 DECISION PENDING — DATA MODEL FORK:
      OPTION A: pair-key as partition key (status quo).
        Days, users, all child tables addressed by pair_key.
        Username/icon change = full migration of every row
        for that pair. O(n_days) work per migration.
      OPTION B: stable pair_id + lookup pair_key.
        Each pair gets a stable UUID (pair_id) at first
        creation. Tables address by pair_id. The pair-key is a
        lookup index, recomputed when username/icon changes,
        but rows don't move. O(1) work per migration.

      RECOMMENDATION: Option B. Separates identity (who you
      are) from addressing (how the database finds you).
      Migrations stop being expensive operations and become
      single-row index updates. The threat model gets cleaner
      too — pair-key becomes a credential string for lookup
      rather than a partition identifier whose properties
      cascade through every table.

      Cost: schema change before tile 1.3 implementation. One
      additional migration (migration_007_pair_id). Refactor
      of existing INSERT/SELECT statements in the worker scaffold
      to address by pair_id.

      This decision is a BLOCKER for tile 1.3 implementation.
      Captured at end of Section 4 in the rewritten doc with a
      NEXT-MORGAN call-out. Resolve at next home-office
      session before any tile 1.3 code lands.

    Session 8 locked week mode as a v1.x post-launch feature.

    v1.0, freeze the schema, then thread a complex feature
