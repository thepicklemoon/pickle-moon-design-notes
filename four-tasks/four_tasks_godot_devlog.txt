BUG FOUND + FIXED (this session, on device): the partner's override
rendered on neither boot nor poll at first. Cause: I updated the
boot ingest (_on_user_loaded) but the periodic POLL uses a separate
handler (_on_poll_completed, a selective merge) that I'd missed —
it ingested self/partner/pair/days/economy but NOT the override
arrays. So overrides could survive a cold boot and then get dropped
on the very next poll. Fixed: _on_poll_completed now ingests both
override sets (server-authoritative, take-as-is, same as boot).
After the fix: partner's overridden Wednesday slot 3 reads "HER WED"
while the other three show her standard four — the mirror confirmed
on screen.

Lesson (logged): when adding a server field the client must hold,
EVERY ingest path needs it — boot (load_user → _on_user_loaded) AND
poll (poll_completed → _on_poll_completed). They're separate
handlers by design (full reset vs selective merge); a field added to
one silently dies on the other.

Also surfaced (→ todo, launch-hardening, NOT this tile): the client
conflates "network unreachable" with "server returned 404 for my
stored user_id." A DB reset wipes the server but the device keeps
its saved id, so boot calls load_user(saved_id), gets nothing, and
parks on "Couldn't reach the server. Tap to retry." — which is why a
reset does NOT route to onboarding (onboarding is gated on local
identity absence, not server absence). Harmless in dev (re-seed) but
in prod a deleted/dangling account traps a real user on infinite
retry. Fix before launch: distinguish 404-on-load (→ onboarding /
"account no longer exists") from a real network error (→ retry).

Time: ~5h ESTIMATE (correct at close).

Closed 4.D2. s38 shipped the server + read-side mirror; this session
built the authoring UI on top, then spent most of its length hunting —
and finally fixing — a sealed-day immutability break that the new
override layer exposed. MODEL A locked (Morgan): a sealed day PRESERVES

SHIPPED — authoring UI (client):
- weekday_edit_dialog.gd (NEW, all-code modal, cloned from the
  tasks_edit_dialog house pattern — scrim 0.6 + centred card). Four
  [checkbox][LineEdit] rows: inactive shows the standard label greyed +
  locked; tick → editable; untick → PAUSE (typed text preserved for a
  re-tick); empty-on-blur → revert to inactive. Save builds the server
  invariant (4 non-empty + mask) and calls the optimistic write. A
  zero-ticked save writes an all-inactive row = CLEARS the weekday's
  override (the un-set path). NO today-disarm (deemed overkill; known
  accepted seam: tick today, override that slot, untick → flips to the
  override, because an unsealed day resolves live).
- State.request_weekday_override(weekday, labels, mask) — OPTIMISTIC:
  find-or-insert the local row, emit self_changed for instant repaint,
  write via Backend, roll back the WHOLE override array on fail. New
  signal weekday_override_failed. Snapshots the whole array (uniform
  insert/update rollback; ≤7 tiny rows — clarity over cleverness). No
  in-flight guard (a mid-write poll stomp self-corrects sub-second, same
  accepted risk as self_user).
- calendar_grid.gd — WeekdayStrip (the MON-SUN static labels that already
  existed in calendar_grid.tscn) long-press HOSTED in the grid: timing in
  _process, drag-cancel past TAP_DRAG_THRESHOLD, LONG_PRESS_DURATION;
  emits weekday_long_pressed(1-7). SELF-GATED via _is_self() (user_id ==
  self_user.user_id) — connections are made on both grids but the handler
  no-ops on the partner grid, so the partner strip is inert. Active-
  override header BOLD = colour step TEXT_WEEKDAY → TEXT_HEADER (no bold
  font variant in Fonts.gd; tunable later, Morgan's call).
- App.gd — opens the editor on weekday_long_pressed, same modal hygiene as
  the hamburger (ceremony-inert, one editor at a time, cross-guarded vs
  _tasks_edit). tasks_edit_dialog.gd — banner when today's weekday has an
  active override (edits there reach only the non-overridden slots).

THE IMMUTABILITY FIX (the session's hard problem):
- BUG: a sealed day showed the LIVE override (e.g. "dsfaf" on a sealed
  past Wednesday). The new override layer exposed a latent gap.
- ROOT (two, conflated): (1) the resolver decided "is this day frozen?"
  by SNAPSHOT-PRESENCE (sealed_labels is 4-elem), not by SEALED-NESS
  (sealed_at != null). A sealed day with an empty '[]' snapshot — a row
  sealed before the sealed_labels column existed, a seed-gap row, or a
  PHANTOM rowless past day (getUserPayload ships every day from
  {year-1}-01-01, so old test rows land in State.days with no real row) —
  fell through to the override. (2) the unsealed/override branch had NO
  past-guard, so today's override also leaked onto unsealed/phantom PAST
  days.
- FIX (State.resolve_labels_for_day rewritten, for_seal:bool=false param):
  (1) gate on sealed_at — a sealed day returns its snapshot, or the
  standard four if the snapshot is empty, but in NO case the override.
  (2) DISPLAY-only past-guard — a past unsealed/phantom day shows the
  standard four (no backward leak); normal operation never hits this (the
  claim seals past days) so it only catches anomalies. (3) for_seal flag —
  _apply_seal_to_day passes true, which SKIPS both the sealed-branch (the
  row is still unsealed at freeze time — we're minting its snapshot) AND
  the past-guard (a walkback day DID live under the override → its memory
  should keep it). MODEL A.
- _apply_seal_to_day REORDERED: resolve the labels WHILE the row is still
  unsealed, THEN stamp sealed_at — else the new sealed-gate would take the
  frozen branch, find no snapshot yet, and freeze the standard four,
  dropping the override.
- SERVER mirror (index.ts): resolveSealedLabels(standardJson, overrides,
  date) + isoWeekdayOf(date) added — the seal snapshot is now the RESOLVED
  four per-day (was the standard four computed once before the loop, the
  4.SL retrofit owed since s37). Applied at ALL THREE seal sites: the
  sealSpan loop, and the gap-rescue breaker INSERT in resolveClaim
  (handledExternally bypasses the loop, resolved separately). Grey-rescue
  rides the loop. devkitWipeStatements already clears the override table,
  so the seed backfill (standard four) stays correct — no seed change. The
  server does NOT past-guard (every sealSpan day is past by construction;
  guarding would mean overrides never freeze, which Model A rejects) — it
  matches the client for_seal path so the two snapshots agree.

VERIFIED (device):
- Editor: long-press a weekday header on the SELF panel opens it;
  activate/type/save → live override renders + header bolds; pause/re-tick
  restores text; empty-on-blur reverts.
- Immutability: phantom past days now show the standard four (was the
  bug); real seeded sealed days show their snapshot.
- SEAL-THROUGH (the gate Morgan wanted closed before dogfooding): set an
  override on today's weekday, DevKit "Today +1 day" → bg/fg the app →

PROCESS NOTE (cost ~3 false bug-hunts before the real fix landed):
autoloads do NOT hot-reload. State.gd edits require Project → Reload
Current Project (NOT stop-scene) before they're live — a running editor

symptom across multiple attempts was a stale on-disk State.gd the whole

when something is "deployed but still broken," disk-verify the CLIENT
file (Select-String for a unique token) and ground-truth the DB FIRST,
theorise after. Three correct fixes sat un-run because the file wasn't
live. Adding a boot version stamp was floated and declined (would be
forgotten).
