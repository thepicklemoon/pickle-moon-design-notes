ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Top-level scene tree:
  Main (Node)
  ├── Backend (Autoload — singleton)        — server I/O, signals
  ├── State (Autoload — singleton)          — in-memory mirror of pair data
  ├── ThemeManager (Autoload — singleton)   — palette swap, theme_changed signal
  ├── Audio (Autoload — singleton, v1.x)    — slot ticks, coin sounds, stamp thump
  ├── Loader (Scene)                        — identity check, pair-key resolve
  ├── Onboarding (Scene)                    — name, tasks, slot intro, emoji pick
  ├── App (Scene)                           — main two-panel UI
  └── DevKit (Scene, debug-only)            — bug inbox, seed tools

Source of truth: server. State is a mirror.
Write flow: UI → State.set_*() → Backend.save() → server confirms
            → State emits signal → UI updates from signal.
Optimistic local updates with rollback on server reject.

Server architecture (locked session 2):
  - Cloudflare Worker (TypeScript) at four-tasks-api.thepicklemoon.workers.dev
  - D1 SQL database (Oceania region) bound as env.DB
  - Four tables: pairs, users, days, bug_reports
  - Pair-key migrations are re-entrant + transaction-wrapped

DEV ENVIRONMENT LESSONS:
  - Windows PowerShell `curl` is an alias for Invoke-WebRequest, NOT
    real curl. Different semantics, security prompt UX, different
    output format. Adopted convention: use `curl.exe` explicitly on
    this project, always. Same flags as everywhere else, same output
    as everywhere else.
  - PowerShell mangles JSON bodies on the command line — backslash-
    escaped quotes don't survive interpolation. Stored JSON in a file
    and used `--data-binary "@file.json"` to sidestep quoting. Adopted
    as the convention for any curl-with-body command going forward.
  - `npx wrangler dev` drifted to remote mode mid-startup twice during
    this session (no clear root cause — possibly hotkey, possibly
    config). Workaround: `npx wrangler dev --local` with the explicit
    flag prevents the drift. Adopted as the convention.
  - Local D1 is a separate SQLite file from remote D1. Tile 1.1 only
    applied schema to remote. First time `--local` ran, `pairs` table
    didn't exist locally and the endpoint 500'd. Fix: apply schema.sql
    with `--local`. Migrations 001/002 are baked into schema.sql so
    re-applying them to a fresh local DB fails with "duplicate column"
    — harmless, skip them for local seed. Will document the fresh-
    clone setup sequence in the /server/README.md when that lands in
    tile 1.3.
  - SQL seed via shell: PowerShell escaped-quote interpolation
    corrupted a JSON string column on first insert attempt (stored
    `\workout\` instead of `"workout"`). Caught immediately because
    the endpoint 500'd on JSON.parse — the try/catch did exactly what
    it was designed for. Fix: put SQL in a .sql file, use `--file=`,
    no shell escaping in play. Convention adopted: anytime SQL needs
    quoted strings, write the SQL to a file.

  Pattern across these: PowerShell's argument handling is a
  consistent source of friction. The fix is the same every time —
  use an explicit executable (curl.exe), use file references over
  inline strings (--data-binary @file, --file=seed.sql). Doing this
  by default removes a class of mystery bugs.
