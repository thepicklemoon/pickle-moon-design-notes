========================
Last updated: May 2026 (session 5, end)

This list is driven by the roadmap (`four_tasks_godot_roadmap.md`). All tiles
from the roadmap are seeded below as `[ ]` items, organised by phase. Tick
items off as they land.

The "next thing" rule: at any moment, the next thing to work on is the
topmost incomplete tile in the current phase. If you don't feel like that
one, drop down to the next, but don't jump phases.

Convention (set session 2): every tile, when ticked, gets a `Time: ~Xh`
or `Time: ~Xmin` entry on its title line. The data informs realistic
estimation for future tiles and tells future-Morgan "do I have time to
start this one." Times are approximate by design — round to the nearest
useful chunk.

Tiers (for ad-hoc items that don't fit the phase structure — bugs found
during the port, design changes, etc.):
  ACTIVE      — ready to implement now.
  PARKED      — known, not blocking, picked up when relevant.
  DEFERRED    — explicitly v1.x or later.

APPTRIOC FORK ARCHITECTURE (locked session 5)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

APPtrioc is no longer "a marketing fork of the web prototype." It is a
SNAPSHOT of the Four Tasks Godot project taken at a specific point in
Phase 4, exported to HTML5, deployed to Cloudflare Pages, configured for
unilateral pairing + atrioc-themed stickers + three-button onboarding,
pointed at its own Worker + D1.

The fork point is the natural divergence between the universal experience
(daily ritual, four-task UI, morning sequence, partner reactions, MOTD,
streaks, basic theming) and the deep theme system (sticker picker
contextual menu, calendar leader, palette source, retint mechanism,
themed sticker catalogue). Tile 4.14 has been SPLIT to make this fork
point explicit in the tile structure.

  Pre-fork: tile 4.14a — basic picker grid, tap-to-toggle pool membership
            ONLY. No long-press, no context menu, no theme system depth.
            APPtrioc inherits everything up to and including this tile.

  Fork:     tile X-series — snapshot Four Tasks, fork APPtrioc, skin,
            ship. Four Tasks development pauses here briefly while
            APPtrioc launches separately.

  Post-fork: tile 4.14b — long-press context menu, theme manager,
             palette source switching, retint mechanism. Four Tasks
             only. APPtrioc never gets these.

LAUNCH SEQUENCE (locked session 5):

  1. Four Tasks ships to both stores at end of Phase 5 (independent
     of APPtrioc).
  2. APPtrioc development + deployment happens in Phase 5b (between
     Four Tasks store submission and the atrioc video drop).
  3. The atrioc video + Reddit post drops AFTER Four Tasks is live
     on stores AND APPtrioc is live at apptrioc.thepicklemoon.com.

This is a sequenced launch, NOT a simultaneous launch. The earlier
"simultaneous launch gated on store approval" plan is superseded.
Reason: APPtrioc IS the Four Tasks codebase up to the fork point;
shipping a polished APPtrioc requires Four Tasks Phase 4 to be
substantially complete first. See four_tasks_marketing_notes.md
for the full strategic rationale and contingency thinking.

The atrioc moment is now a FORCE MULTIPLIER on an already-shipped
product, not a launch driver. Four Tasks launches on its own merits
via subculture seeding; APPtrioc + the video amplify whatever has
already taken hold.

    PRE-CODE LOCKS (session 2):
      • Response envelope: uniform { "ok": true, "data": ... } on
        success, { "ok": false, "error": "code", "message": "..." }
        on failure. Every endpoint, every time.
      • HTTP status codes: 200 success, 400 bad request, 404 not
        found / unauthorised, 409 conflict (stale pair-key /
        collision), 500 server error. No 201/202/204/401/403/429.
      • Auth: caller named in URL. Self-writes have the caller as
        the path segment. Cross-user writes use ?caller=name query
        param. See tile 1.3 for the full defensive logic flow.
    SESSION 3 NOTES:
      • /resolve URL corrected from POST /pair/:key/resolve to
        POST /resolve. The caller doesn't have a key yet — that's
        the whole point of resolve. Body shape:

      • Day history cap on GET /pair/:key: calendar year + previous
        year. Older days remain in DB but aren't returned. See new
        keepsake-export note in ACTIVE.
      • Server-side input validation in /resolve enforces the
        canonical-form rules (no '|', no '__', non-empty strings).
        Onboarding will mirror these rules client-side at tiles

    AUTH PATTERN (locked tile 1.2 session 2): every request names
    the caller in the URL. Self-writes: the path segment IS the
    caller. Cross-user writes (partner reactions): append
    ?caller=name. Server defensive logic:
      1. Parse target user from URL path
      2. Parse caller from query param, defaulting to target
      3. Verify caller is one of the two users in this pair (404 if not)
      4. Apply field-level rules per caller's role (404 on violation)
      5. Apply immutability / state checks (409 on state violation,
         404 on permission violation)
      6. Write in a transaction
    See the design doc for the full per-endpoint rule tables.

[ ] curl-test convention for the project — use curl.exe explicitly.
    Captured session 3. On Windows PowerShell, `curl` is an alias
    for Invoke-WebRequest, which has different semantics, a
    security-prompt UX, and a different output format. Using
    `curl.exe` invokes the real curl that ships with Windows
    10/11. Standard curl flags work, output matches documentation
    everywhere else, copy-paste from tutorials works without
    translation. Adopted as the project convention going forward.
    Also: PowerShell mangles JSON bodies on the command line
    (backslash-escaped quotes don't survive). For any curl with
    a JSON body, write the body to a file and use --data-binary
    "@filename.json". Sidesteps the quoting entirely. The /server/
    README.md (when written in tile 1.3) should follow these
    conventions in its examples.
