[ ] 1.2 [CODE] Worker endpoints v1.
    GET /pair/:key, PUT /pair/:key/users/:name,
    PUT /pair/:key/users/:name/day/:date, POST /pair/:key/resolve,
    POST /bug_report, GET /leaderboard. All with input validation. All
    returning consistent JSON envelopes.
    PRE-CODE LOCKS (session 2):
      • Response envelope: uniform { "ok": true, "data": ... } on
        success, { "ok": false, "error": "code", "message": "..." }
        on failure. Every endpoint, every time.
      • HTTP status codes: 200 success, 400 bad request, 404 not
        found / unauthorised, 409 conflict (stale pair-key /
        collision), 500 server error. No 201/202/204/401/403/429.
      • Auth: caller named in URL. Self-writes have the caller as
        the path segment. Cross-user writes use ?caller=name query
        param. See tile 1.3 for the full defensive logic flow.
    NOTE: /resolve endpoint takes the six-value tuple, derives the
    pair-key, looks up the pair. See pair_key_v2 design notes for
    canonical-form rules.
    SESSION SHAPE: realistic split is read endpoints first (GET /pair/:key,
    POST /pair/:key/resolve, GET /leaderboard) one session, write
    endpoints (PUT user, PUT day, POST bug_report) another. Cleaner
    per-session scope, each verifiable end-to-end with curl before
    moving on.

[ ] 1.3 [CODE] Worker defensive write rules.
    "Never reduce task count" rule. "Only this user can write to their own
    slot in the pair" rule (verified by passing pair_key + own_name on
    every write). curl tests documented in /server/README.md.
    AUTH PATTERN (locked tile 1.2 session 2): every request names the
    caller in the URL. For self-writes, the path segment IS the caller.
    For cross-user writes (e.g. partner reactions), append ?caller=name
    as a query param. Server defensive logic:
      1. Parse target user from URL path
      2. Parse caller from query param, defaulting to target
      3. Verify caller is one of the two users in this pair (404 if not)
      4. Apply field-level rules per caller's role (404 on violation)
      5. Apply immutability checks (404 on second write to non-NULL
         reaction column)
      6. Write in a transaction
    No 403s — "permission denied" semantics fold into 404.
    PLUS: pair-key migration logic (transactional, re-entrant) for
    username/icon changes. See pair_key_v2 design notes section
    "Migration semantics" for the full algorithm.
    PLUS: stale-pair-key write handling. Server responds HTTP 409 with
    {"error":"stale_pair_key","current_key":"<hex>"} when a write
    addresses a key that's been migrated away. Client updates its
    cached key from the response and retries. See v2 notes
    "Stale-key write handling".
    PLUS: FIELD-LEVEL write rules on `days` for partner reactions.
    Reactions are post-seal-only AND immutable-once-placed:
      • Live day (sealed_at IS NULL): owner writes own fields,
        nobody writes reaction columns
      • Sealed day (sealed_at IS NOT NULL): owner writes nothing,
        partner writes reaction columns ONLY if the column is
        currently NULL. Reject second writes to non-NULL reaction
        columns with 404 (server-enforced immutability).
    Cross-role / cross-phase / immutability violations all reject with 404
    (no 403s in our model). See
    four_tasks_partner_reactions_design_notes.md "Field-level write
    rules" for the full lookup table.
